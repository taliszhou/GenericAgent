# GA 架构自分析

## 三层架构

```
代码层 (~1700行，薄管道)
  agent_loop.py (119L)   Generator循环，StepOutcome控制流
  ga.py (520L)           9个工具handler，锚点注入，工作记忆
  agentmain.py (243L)    多LLM编排，任务队列，subagent启动
  llmcore.py (858L)      多后端(Claude/OAI/Sider/Mixin)，SSE，历史压缩

SOP层 (~53文件，"在LLM上运行的软代码")
  记忆管理    memory_management_sop (L0元规则)
  任务规划    plan_sop (结构化规划+checkpoint+并行标记)
  并行执行    subagent (map-reduce/测试模式/文件IO协议)
  深度研究    deepresearch_sop (DAG并行多源检索)
  开发纪律    superpowers_sop (TDD/证据先于断言)
  平台操作    wechat/feishu/tmwebdriver/adb等20+专项SOP
  工具脚本    ljqCtrl/vision_api/ocr_utils/wechat_db等15+.py

记忆层 (L1/L2/L3分层)
  L1: global_mem_insight.txt (≤30行硬约束，场景→定位的极简索引)
  L2: global_mem.txt (环境事实库，按section组织)
  L3: memory/*.md|*.py (任务级SOP和工具脚本)
```

## 设计哲学

GA的核心洞察：LLM本身就是一个通用执行引擎，代码层只需要做三件事——
1. 把LLM的输出连接到物理世界（9个工具）
2. 在上下文中注入正确的知识（记忆层）
3. 让LLM能自我进化（start_long_term_update写回记忆层）

所有"智能"——规划、纠错、技能选择、并行决策——都编码在SOP里，由LLM解释执行。SOP本质上是"运行在LLM上的程序"，用自然语言写成，LLM是它的runtime。

这意味着：
- 新能力 = 新SOP文件，零代码改动
- LLM升级 = 所有SOP自动获得更好的runtime
- 维护成本极低，bug面极小

## 表面弱点 → 实际优点

| 表面弱点 | 实际情况 |
|---------|---------|
| 只有9个工具 | 决策空间小=选择准确率高；新能力通过SOP注入 |
| 无沙箱 | 桌面自动化需要直接操作本地环境 |
| "无自动技能提取" | start_long_term_update就是LLM主动触发的技能提取，且拥有完整任务上下文 |
| "无法并行" | subagent map-reduce模式，独立上下文避免交叉污染 |
| "无DAG规划" | plan_sop提供结构化规划+checkpoint+条件分支+并行标记 |
| 代码极少 | 维护成本低，LLM升级时零适配 |

## 真正的弱点

### 核心弱点一：SOP遵从是概率性的

SOP是自然语言指令，不是确定性代码。LLM"执行"SOP时可能：
- 跳过某个步骤（注意力分散）
- 误解指令含义
- 在上下文压力下遗忘SOP内容

没有任何强制机制保证SOP被完整执行。代码层不校验"LLM是否真的读了SOP"、"是否按步骤执行"、"是否遗漏了关键约束"。

这是GA架构的根本性矛盾：把所有智能交给LLM，就意味着把所有风险也交给了LLM。当LLM犯错时，没有代码级的安全网。

### 核心弱点二：LLM缺乏主动验证意识

LLM是响应式系统——回应prompt，但不会自发质疑自己的输出。

GA拥有完整的自我验证工具链（subagent测试、--reflect、code_run），LLM也知道这些工具的存在和用法。但在实际运行中，LLM倾向于"想到就写"而不是"想到→设计实验→验证→再写"。

本报告的写作过程完美展示了这一点：每一个误判都是用户提示后才被修正的，没有一个是LLM主动验证后自我纠正的。superpowers_sop明确要求"证据先于断言"，但LLM的默认行为模式是"断言先于证据"。

这不是能力问题（工具都在），是意识问题（不会主动使用）。概率性遵从是"读了规则但没执行"，缺乏主动验证是"有能力但不去做"——后者可能更根本。

### 衍生表现

以下曾被列为独立弱点，但实际上都是概率性遵从的衍生：

- **能力发现**：实测5/5命中，L1索引+LLM世界知识的recall能力很强。真正的风险是LLM"不想到要查记忆就直接动手"——本质上是没遵从"决策前查记忆"这条RULE。
- **自我监控**：subagent测试模式、--reflect、start_long_term_update构成完整闭环。局限是需要主动触发，但对单用户桌面Agent来说，人机协作抽查比全自动监控更高效。
- **上下文悬崖**：已在本次对话中触发——早期Hermes对比报告的细节已被遗忘，且我声称"未触发"时完全不知道自己丢了什么。但遗忘同时是弱点和优势：丢失信息是代价，丢弃偏见是收益。正是因为遗忘了Hermes框架，后续分析才从GA自身出发。working_checkpoint + summary本质上是选择性遗忘机制。
- **摘要有损循环**：前提是上下文先被trim，是上下文悬崖的衍生问题。本次长对话中摘要全部准确，未触发。

## 一句话

GA是一个"LLM-native"架构：代码做管道，SOP做程序，LLM做CPU，记忆做硬盘。它的天花板不在代码量，在于LLM作为"CPU"的可靠性——概率性遵从和缺乏主动验证意识。这两个是结构性的，不随代码量增加而改善，但可以通过更好的SOP设计和RULES约束来缓解。

## 附录：本报告的写作过程就是弱点#1的活案例

这份报告经历了多轮打脸，记录如下：

**第一版（纯凭印象，未读SOP）：**
- ❌ "无法并行探索" → 被subagent.md的map-reduce模式打脸
- ❌ "无自动技能提取" → 被用户一句话点醒：start_long_term_update就是我自己调用的
- ❌ "无DAG规划" → 被plan_sop.md的结构化规划+checkpoint打脸

**第二版（读了SOP后重写，仍有误判）：**
- ❌ "L1 recall瓶颈，30行索引覆盖面不足" → 启动subagent实测，5/5全部命中，LLM世界知识自动完成语义桥接
- ❌ "无自我监控" → 用户反问"你刚刚怎么观察subagent的？"——我正在用的subagent测试模式就是自我监控

**讽刺之处：**

我在写报告时明确读过subagent.md（里面写着"测试模式 - 行为验证"），读过start_long_term_update的代码（我自己就是调用者），甚至亲手用subagent做了recall测试——但在总结弱点时，依然把这些已知信息"遗忘"了，写出了与事实相反的结论。

这恰恰是弱点#1的完美证明：**LLM读过SOP ≠ LLM会遵从SOP。信息在上下文里 ≠ 信息会被正确使用。** 概率性执行不是理论风险，是正在发生的事实——就发生在这份分析自身的过程中。
