# 开源项目调研报告 SOP

> **输入**：项目名 或 GitHub URL
> **输出**：docs/reports/项目名_调研报告.md

> **适用范围**：当需要研究一个新的开源项目或技术产品时

---

## Step 1: 收集信息

### 1.1 基本信息（5分钟内完成）
- [ ] Star / Fork / Watch 数
- [ ] 最近更新时间、最后 Release 版本
- [ ] 贡献者数量
- [ ] README 一句话描述

### 1.2 业务纵向分析
- [ ] 开源时间、创始人/公司背景
- [ ] 关键版本发布节点（v1.0、v2.0...）
- [ ] 发展轨迹与里程碑
- [ ] 融资情况（若有）

### 1.3 技术纵向分析 ⭐
- [ ] 编程语言及版本
- [ ] 核心模块/目录结构概览
- [ ] 架构风格（单体/微服务/插件化/云原生...）
- [ ] 主要依赖库及版本新旧程度
- [ ] 设计模式或关键技术选型
- [ ] CI/CD 配置、测试覆盖率
- [ ] 文档完整性（API docs/开发指南/示例）

### 1.4 竞品横向分析
- [ ] 直接竞品列表（≥3个）
- [ ] Star 数/社区规模对比
- [ ] 技术栈差异对比
- [ ] 差异化定位总结

### 1.5 决策还原
- [ ] 技术选型原因推测
- [ ] 商业模式/盈利方式
- [ ] 扩展性设计分析
- [ ] 战略方向合理性评估

### 1.6 技术风险评估
- [ ] 依赖安全漏洞情况
- [ ] 技术债务（过时依赖/未处理 Issue）
- [ ] 维护状态（响应速度/活跃度）

---

- 一句话定义
- 纵向分析（起源与背景、发展轨迹、关键里程碑）
- 横向分析（竞品对比、生态位定位）
- **技术架构分析**（源代码结构、技术栈、架构设计评估、代码规模）
- 未来预测（最可能/最危险/最乐观剧本）
- 资源清单（Star 数、贡献者、代码仓库等）

**产出物**：报告文件（存放于 `/Users/taliszhou/wiki/source/` 目录）

## Step 2: 生成报告

将信息整理为标准化报告，格式如下：

- **一句话定义**：用最简洁的话描述项目是什么
- **业务分析**：纵向（起源/背景/发展轨迹）+ 横向（竞品/生态位）
- **技术分析**：架构/技术栈/工程实践/风险
- **未来预测**：最可能 / 最危险 / 最乐观剧本
- **资源清单**：Star 数、贡献者、仓库地址、数据来源链接

**产出物**：存放于 `docs/reports/` 目录

---

## Step 3: 交付

1. 报告存入 `docs/reports/项目名_调研报告.md`
2. 直接展示给用户，包含数据来源说明
3. 告知用户是否需要执行知识库入库 SOP

---

## 中止条件

- 项目已归档（>1年无更新）→ 报告中标注风险，继续完成
- 文档严重缺失 → 中止并告知用户
- 私有仓库/无权限 → 中止

---

## 注意事项

- 本 SOP 完成后，如用户同意，可继续执行《知识库入库 SOP》
- 不要跳过后续的资料页/综合页/交叉链接等步骤
